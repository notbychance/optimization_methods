# Optimization Methods Toolkit

Набор независимых консольных Python-модулей для решения задач оптимизации, сетевого анализа, динамического программирования, теории игр и целочисленного программирования.

Проект устроен как общий запускатор и набор отдельных модулей. Каждый модуль читает входные данные из JSON-файла, выполняет расчёт собственным алгоритмом без библиотек оптимизации и сохраняет пошаговый протокол решения в Markdown-файл.

## Возможности

- единый запуск модулей из корня проекта через `main.py`;
- отдельный запуск каждого модуля из его каталога;
- входные данные в формате JSON;
- экспорт промежуточных таблиц, матриц, графов, деревьев и итогов в Markdown;
- реализация алгоритмов на стандартной библиотеке Python;
- использование точных рациональных чисел через `fractions.Fraction` там, где это важно для симплексных и матричных вычислений.

## Состав модулей

| Номер | Каталог | Назначение |
|---:|---|---|
| 1 | `lab1_linear_programming` | Линейное программирование: графический метод, угловые точки и график области решений |
| 2 | `lab2_simplex` | Линейное программирование: двухфазный симплекс-метод |
| 3 | `lab3_dual_simplex` | Линейное программирование: двойственный симплекс-метод и матричное представление базиса |
| 4 | `lab4_transportation` | Транспортная задача: начальный план и оптимизация методом потенциалов |
| 5 | `lab5_network_models` | Сетевые задачи: MST, кратчайший путь, максимальный поток, поток минимальной стоимости, проверка предвыигрышной конфигурации поля |
| 6 | `lab6_dynamic_programming` | Детерминированное динамическое программирование: рюкзак, конечная многоэтапная модель, мультипликативное ограничение |
| 7 | `lab7_parametric_linear_programming` | Параметрическое линейное программирование с параметром в целевой функции или правых частях ограничений |
| 8 | `lab8_game_theory` | Матричные игры, критерии принятия решений, биматричные игры, дерево решений профилактического обслуживания |

Каждый каталог содержит локальный `README.md`, файл запуска `main.py`, модули алгоритма, модуль чтения JSON, модуль экспорта снимков и каталог `examples`.

## Требования

Нужен Python 3.10 или новее.

Проверка версии:

```bash
python --version
```

Сторонние зависимости не требуются.

## Общий запуск из корня проекта

Формат команды:

```bash
python main.py --number НОМЕР --input ВХОДНОЙ_ФАЙЛ.json --output РЕЗУЛЬТАТ.md
```

Короткая форма:

```bash
python main.py -n НОМЕР -i ВХОДНОЙ_ФАЙЛ.json -o РЕЗУЛЬТАТ.md
```

Параметры:

| Параметр | Короткая форма | Описание |
|---|---|---|
| `--number` | `-n` | Номер модуля: от 1 до 9 |
| `--input` | `-i` | Путь к входному JSON-файлу |
| `--snapshot`, `--output`, `--export` | `-s`, `-o` | Путь к Markdown-файлу результата |

Если параметры не переданы, общий запускатор запросит их интерактивно.

### Как ищется входной файл

Для относительного пути общий запускатор проверяет несколько мест:

1. путь относительно корня проекта;
2. путь относительно каталога выбранного модуля;
3. файл с указанным именем в каталоге `examples` выбранного модуля.

Благодаря этому можно указывать как полный путь, так и только имя файла из `examples`.

## Примеры общего запуска

Перед запуском можно создать каталог для результатов:

```bash
mkdir -p results
```

Для PowerShell:

```powershell
New-Item -ItemType Directory -Force results
```

Команды:

```bash
python main.py -n 1 -i lab1_variant17.json -o results/lab1_variant17.md
python main.py -n 2 -i lab2_example.json -o results/lab2_simplex.md
python main.py -n 3 -i lab3_dual_example.json -o results/lab3_dual_simplex.md
python main.py -n 4 -i lab4_transport_example.json -o results/lab4_transportation.md
python main.py -n 5 -i lab5_tictactoe_concrete_example.json -o results/lab5_network_models.md
python main.py -n 6 -i lab6_multiplicative_concrete_example.json -o results/lab6_dynamic_programming.md
python main.py -n 7 -i lab7_rhs_parameter_example.json -o results/lab7_parametric_lp.md
python main.py -n 8 -i lab8_decision_example.json -o results/lab8_game_theory.md
python main.py -n 9 -i lab9_integer_example.json -o results/lab9_integer_lp.md
```

Полный набор команд приведён в `run_all_labs_commands.md`.

## Запуск отдельного модуля напрямую

Каждый модуль можно запускать отдельно из его каталога.

Пример:

```bash
cd lab2_simplex
python main.py -i examples/lab2_example.json -s simplex_snapshot.md
```

Ещё один пример:

```bash
cd lab5_network_models
python main.py -i examples/lab5_tictactoe_concrete_example.json -s network_snapshot.md
```

## Входные данные

Все входные данные задаются в JSON. Формат зависит от модуля:

- модули 2, 3, 7 и 9 используют целевую функцию, переменные и ограничения;
- модуль 4 использует запасы, потребности и матрицу стоимостей;
- модуль 5 использует список вершин и рёбер либо поле для проверки конфигурации;
- модуль 6 использует описание дискретной или аналитической модели динамического программирования;
- модуль 8 использует платёжные матрицы, таблицы решений, пары матриц выигрышей или параметры дерева решений.

Подробные схемы входных файлов описаны в локальных README соответствующих модулей.

## Файлы результата

Markdown-файл результата обычно содержит:

- исходные данные в обработанном виде;
- промежуточные таблицы, матрицы или структуры алгоритма;
- описание выбранных переходов, ведущих элементов, базисов, циклов, путей или ветвей;
- итоговый статус;
- найденное решение или причину, по которой решение не получено.

## Типовые статусы

| Статус | Смысл |
|---|---|
| `optimal` | Найдено оптимальное решение |
| `infeasible` | Система ограничений несовместна |
| `unbounded` | Целевая функция не ограничена на допустимой области |
| `not_dual_feasible` | Начальная таблица не подходит для двойственного симплекс-метода |
| `input_required` | Входной файл прочитан, но для расчёта не хватает конкретных данных |
| `prewinning` | Для поля найдён ход, создающий выигрышную цепочку |
| `not_prewinning` | Предвыигрышные ходы не найдены |
| `optimal_cycle_found` | Найден оптимальный цикл профилактического обслуживания |

## Рекомендации по добавлению своих данных

1. Скопируйте ближайший пример из каталога `examples` нужного модуля.
2. Замените коэффициенты, ограничения, матрицы, граф или параметры модели.
3. Запустите модуль через корневой `main.py` или локальный `main.py`.
4. Проверьте консольный статус и Markdown-файл результата.

## Отладка запуска

### Не найден входной файл

Проверьте путь после `--input`. Можно передать абсолютный путь, путь относительно корня проекта, путь относительно каталога модуля или имя файла из `examples`.

### Не создан файл результата

Проверьте путь после `--output` или `--snapshot`. Родительская папка создаётся автоматически при общем запуске. При локальном запуске лучше заранее создать папку назначения.

### Неподдерживаемый тип задачи

Проверьте поле `problem_type`, `task_type` или `mode` во входном JSON. Допустимые значения перечислены в README нужного модуля.

## Архитектура общего запускатора

Корневой `main.py` не импортирует внутренние файлы модулей напрямую. Он запускает локальный `main.py` выбранного каталога отдельным процессом. Такой подход сохраняет независимость модулей и предотвращает конфликты между одноимёнными файлами вроде `problem_reader.py`, `snapshot.py` и `linear_problem.py`.
