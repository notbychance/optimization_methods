# Модуль 5: сетевые модели, проверка конфигурации поля и сетевое планирование

Консольная программа для решения задач на графах, проверки предвыигрышной конфигурации клеточного поля и расчёта временных характеристик сетевого графика. Все алгоритмы реализованы вручную без специализированных библиотек.

## Поддерживаемые типы задач

| `problem_type` | Назначение | Алгоритм |
|---|---|---|
| `mst` | минимальное остовное дерево | алгоритм Краскала |
| `shortest_path` | кратчайший путь от источника к цели | алгоритм Дейкстры |
| `max_flow` | максимальный поток | алгоритм Эдмондса-Карпа |
| `min_cost_flow` | поток минимальной стоимости | последовательный поиск кратчайших увеличивающих путей по стоимости |
| `prewinning_tictactoe_configuration` | проверка, можно ли одним ходом получить цепочку заданной длины | перебор пустых клеток и проверка направлений |
| `project_scheduling` | сетевое планирование, расчёт ранних/поздних сроков, резервов и критического пути | метод критического пути |
| `critical_path` | псевдоним для `project_scheduling` | метод критического пути |

## Общий формат графовой задачи

```json
{
  "problem_type": "shortest_path",
  "directed": true,
  "nodes": ["A", "B", "C"],
  "edges": [
    {"from": "A", "to": "B", "weight": 4},
    {"from": "B", "to": "C", "weight": 2},
    {"from": "A", "to": "C", "weight": 10}
  ],
  "source": "A",
  "target": "C"
}
```

Поля:

| Поле | Описание |
|---|---|
| `problem_type` | тип задачи |
| `directed` | ориентированный граф; для `mst` обычно `false` |
| `nodes` | список вершин; если не задан, собирается из рёбер |
| `edges` | список рёбер или дуг |
| `from`, `to` | начальная и конечная вершина ребра |
| `weight` | вес ребра для MST и кратчайшего пути |
| `capacity` | пропускная способность для потоковых задач |
| `cost` | стоимость единицы потока для `min_cost_flow` или стоимость работы для `project_scheduling` |
| `source` | источник для путей и потоков |
| `target` | сток или целевая вершина |
| `desired_flow` | требуемый объём потока для `min_cost_flow`; необязательно |

## Формат задачи сетевого планирования

Новый режим работы задаётся так:

```json
{
  "problem_type": "project_scheduling",
  "nodes": ["1", "2", "3", "4", "5", "6", "7"],
  "edges": [
    {"from": "1", "to": "2", "duration": 2, "cost": 11},
    {"from": "1", "to": "3", "duration": 3, "cost": 12},
    {"from": "1", "to": "4", "duration": 8, "cost": 9},
    {"from": "2", "to": "5", "duration": 8, "cost": 30},
    {"from": "3", "to": "5", "duration": 6, "cost": 24},
    {"from": "3", "to": "6", "duration": 12, "cost": 30},
    {"from": "4", "to": "6", "duration": 10, "cost": 30},
    {"from": "5", "to": "7", "duration": 6, "cost": 25},
    {"from": "6", "to": "7", "duration": 3, "cost": 15}
  ]
}
```

Для каждой работы:

| Поле | Описание |
|---|---|
| `from` | начальное событие работы |
| `to` | конечное событие работы |
| `duration` | продолжительность работы |
| `cost` | стоимость работы |

Дополнительно можно задать:

| Поле | Описание |
|---|---|
| `start_event` | начальное событие, если его нужно зафиксировать вручную |
| `finish_event` | завершающее событие, если его нужно зафиксировать вручную |
| `schedule_mode` | текстовое описание режима, например `срочный режим работ` |
| `duration_unit` | единица измерения времени, например `дни` |

Режим `project_scheduling` рассчитывает:

1. топологический порядок событий;
2. ранние сроки наступления событий;
3. поздние сроки наступления событий;
4. резервы событий;
5. раннее и позднее начало/окончание работ;
6. полный и свободный резерв каждой работы;
7. критические работы;
8. критический путь;
9. продолжительность комплекса работ;
10. суммарную стоимость работ.

Для файла `examples/lab5_project_scheduling_variant17.json` результат должен быть:

```text
Продолжительность комплекса работ: 21
Критический путь: 1 -> 4 -> 6 -> 7
Суммарная стоимость работ: 186
```

## Формат задачи проверки поля

```json
{
  "problem_type": "prewinning_tictactoe_configuration",
  "win_length": 5,
  "symbols": {
    "first_player": "X",
    "second_player": "O",
    "empty_cell": "."
  },
  "board": [
    ".......",
    "..XXXX.",
    "...O...",
    "...O...",
    "...O...",
    "......."
  ]
}
```

Для поля поддерживаются три формы:

```json
"board": ["..X", ".O.", "..."]
```

```json
"board": [[".", ".", "X"], [".", "O", "."], [".", ".", "."]]
```

```json
"board": "..X\n.O.\n..."
```

Если `board` передан текстовым описанием, а не конкретной матрицей, программа завершится статусом `input_required`.

## Результат

В консоль выводятся тип задачи, статус, сообщение и итоговые данные: путь, выбранные рёбра, поток, список найденных ходов или временные характеристики сетевого графика. Markdown-файл содержит исходные данные, промежуточные шаги и итог.

## Возможные статусы

| Статус | Смысл |
|---|---|
| `optimal` | задача решена |
| `no_path` | путь между заданными вершинами не найден |
| `unreachable` | целевая вершина недостижима |
| `disconnected` | граф несвязный для построения остова |
| `infeasible` | требуемый поток построить невозможно |
| `input_required` | для проверки поля не задана конкретная матрица |
| `prewinning` | найден хотя бы один выигрышный ход |
| `not_prewinning` | выигрышные ходы не найдены |
